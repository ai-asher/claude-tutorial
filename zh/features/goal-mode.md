# 目标模式 /goal

平时和 Claude 是"一问一答"——它做完一轮就停下等你。**目标模式（`/goal`）**反过来：你给它一个**完成条件**，它会跨越多个轮次自己持续工作，直到条件满足才停。

## 什么是目标模式

`/goal` 让你设定一个明确的"做完的标准"，然后 Claude 会自主循环：执行 → 检查是否达标 → 没达标就继续，直到目标完成。

```bash
> /goal 所有测试通过，且 lint 没有 error

Claude:
  🎯 目标已设定：所有测试通过 + lint 无 error
  ▸ 运行测试… 3 个失败
  ▸ 修复 src/auth.ts… 重新测试… 1 个失败
  ▸ 修复 src/utils.ts… 重新测试… 全部通过
  ▸ 运行 lint… 2 个 error
  ▸ 修复… lint 通过
  ✅ 目标达成（4 轮 · 6m 12s）
```

它不会每修一个就停下问你"要继续吗"——只要目标没达成，它就一直推进。

## 怎么用

```bash
# 设定目标
/goal <完成条件>

# 例子
/goal 把 README 翻译成英文，并通过 markdown lint
/goal 重构 payment 模块，保持所有现有测试通过
/goal 找到并修复导致登录超时的 bug，加一个回归测试
```

设定后会出现一个**实时面板**，显示进度：

| 面板字段 | 含义 |
|----------|------|
| elapsed | 已用时长 |
| turns | 已经过的轮次 |
| tokens | 累计 token 消耗 |

## 适合什么任务

目标模式的精髓是"**有明确的、可自动验证的完成标准**"：

| ✅ 适合 | ❌ 不适合 |
|--------|----------|
| 测试全绿、lint 无 error | "把代码写得更优雅"（标准模糊） |
| 把覆盖率提到 80% | "看看有没有可以改进的"（开放式） |
| 修到某个 bug 不再复现 | 需要你频繁拍板的探索性工作 |
| 批量转换直到全部完成 | 涉及不可逆操作（删库、发布） |

::: tip 完成条件要"可验证"
目标越具体、越能用命令验证（跑测试、跑 lint、grep 某个输出），`/goal` 越好用。模糊的目标（"更好""更快"）它无法判断何时算达成。
:::

## 与其他功能配合

### /goal × Plan Mode

复杂任务可以先用 Plan Mode 规划路径，再用 `/goal` 设定终点让它自动跑完：

```bash
> /plan 设计从 REST 迁移到 GraphQL 的步骤
# 审阅方案、批准
> /goal 完成迁移，所有原有 API 测试改写为 GraphQL 后通过
```

### /goal × Auto Mode

[Auto Mode](/zh/features/auto-mode) 会按子任务难度自动选模型，配合 `/goal` 跨轮次工作时，简单步骤用快模型、难步骤用强模型，整体更省。

### /goal × 后台运行

`/goal` 在交互式、`-p` 管道模式、以及后台会话里都能用。长目标可以丢后台跑：

```bash
claude --bg -p "/goal 给所有 service 层函数补测试，覆盖率到 85%"
# 稍后用 claude agents 查看进度
```

::: warning 设好边界
目标模式会自主跑很多轮，务必：
- 给**可验证**的明确条件，避免它"永远觉得没达标"
- 配合 `--max-turns` 和 `--max-budget-usd` 设上限
- 不要给涉及不可逆操作的目标
:::

## 小结

| 要点 | 记忆点 |
|------|-------|
| 一句话定义 | 设定完成条件，Claude 跨轮次自动做到达成 |
| 启动 | `/goal <完成条件>` |
| 实时面板 | elapsed / turns / tokens |
| 最适合 | 有可自动验证的明确标准 |
| 安全边界 | `--max-turns` + `--max-budget-usd` |
| 可后台跑 | 配合 `--bg` + `claude agents` |
