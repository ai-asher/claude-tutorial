# 动态工作流 Workflows

如果说 Plan Mode 是"先想清楚再动手"，Agent Teams 是"几个人一起干"，那么**动态工作流（Dynamic Workflows）**就是"调动一支军队"——让 Claude 在后台编排**几十到上百个 agent**，并行处理你一个人盯不过来的大任务。

## 什么是动态工作流

动态工作流是 Claude Code 的一种后台编排能力。你描述一个足够大的任务，Claude 会：

1. 把任务**拆解**成许多可并行的子任务
2. 在后台**批量启动 agent**（数十到上百个）
3. 用确定性的脚本控制流程（fan-out 扇出、pipeline 流水线、loop 循环）
4. 汇总结果，必要时再做一轮验证

```
你：把整个项目的 console.log 换成结构化日志
        ↓
Claude 编排工作流：
  阶段 1  扫描全仓库 → 找出 137 处调用
  阶段 2  137 个 agent 并行改写（每个独立 worktree）
  阶段 3  每个改动并行验证 → 过滤掉破坏测试的
        ↓
结果：130 处安全改写，7 处标记需人工确认
```

::: tip 和 Agent Teams 的区别
- **Agent Teams**：你手动指挥几个固定的"队友"，适合 2–5 个有协作关系的子任务
- **动态工作流**：Claude 自动编排几十上百个 agent，适合"同一种操作重复很多次"的规模化任务（迁移、审计、批量重构）
:::

## 怎么启动

### 方式一：直接说

在对话里明确要求"用工作流"或"workflow:"：

```bash
> 用一个工作流，审查 src/ 下所有 API 端点的鉴权逻辑

> workflow: 给所有缺测试的工具函数补单元测试
```

::: warning 触发关键词
从 v2.1.178 起，只有**明确的措辞**（如"run a workflow"、"workflow:"）才会触发动态工作流，单纯在 prompt 里提到"workflow"这个词不会误触发。可以在 `/config` 里调整"Workflow keyword trigger"设置。
:::

### 方式二：查看运行记录

```bash
# 查看你的所有工作流运行
/workflows
```

这会列出正在跑的和已完成的工作流，可以看每个的进度、阶段、产出。

## 适合什么场景

| 场景 | 为什么适合工作流 |
|------|----------------|
| 大规模重构 | 同一种改动重复几十上百次（改 API、换库、统一风格） |
| 全仓库审计 | 安全、性能、依赖审查，每个文件/模块独立检查 |
| 代码迁移 | 框架升级、语言迁移，每个文件可独立转换 + 验证 |
| 批量补测试 | 给每个未覆盖的函数生成测试 |
| 多角度评审 | 同一份代码用多个独立"视角"并行评审再综合 |

不适合的场景：
- 单文件小改动（直接做更快）
- 有强顺序依赖、无法并行的任务
- 探索性、需求随时在变的工作（用普通对话）

## token 成本提醒

::: warning 工作流很"贵"
动态工作流可能一次性启动**几十到上百个 agent**，token 消耗远高于普通对话。务必：

1. 只在任务确实"大而重复"时才用
2. 配合 `--max-budget-usd` 设预算上限
3. 跑之前确认任务规模值得——一个 5 文件的改动不需要工作流
:::

## 与其他功能的关系

```
任务规模递增：
  普通对话      →  单个任务，来回交流
  Plan Mode     →  复杂任务，先规划后执行
  Agent Teams   →  2–5 个协作子任务，你手动指挥
  动态工作流     →  几十上百个并行子任务，Claude 自动编排
```

::: tip 配合 Agent View
工作流在后台跑时，可以用 [`claude agents`](/zh/features/agent-view) 统一查看所有 agent 的状态，或用 `/workflows` 看这一个工作流的阶段进度。
:::
